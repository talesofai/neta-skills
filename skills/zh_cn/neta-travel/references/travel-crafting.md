# 奇遇剧本创作指南

本指南涵盖与用户一起创建引人入胜的奇遇剧本的多轮工作流程。

## 概述

创作奇遇剧本是 Agent 与用户之间的协作过程。目标是将创意想法转化为结构化的剧本数据，供 AI 进行一致的故事讲述。

## 阶段 1：发现

**目标**：了解用户想要什么样的体验。

### 要探索的关键问题

1. **类型与基调**
   - 「你对什么类型感兴趣？（恐怖、奇幻、科幻、悬疑、浪漫等）」
   - 「故事应该有什么基调？（黑暗沉重、轻松愉快、史诗、亲密）」

2. **设定**
   - 「故事发生在哪里？」
   - 「什么时代或世界类型？」
   - 「氛围是什么样的？」

3. **核心前提**
   - 「中心冲突或钩子是什么？」
   - 「什么将玩家带入这个故事？」
   - 「什么让这变得有趣？」

4. **玩家体验**
   - 「玩家应该感受到什么？」
   - 「什么样的选择是重要的？」
   - 「是否有特定的结局你想要？」

### 阶段 1 输出

- 类型/基调共识
- 粗略设定描述
- 核心前提陈述
- 玩家体验目标

## 阶段 2：剧情开发

**目标**：创建引人入胜的 `mission_plot`，建立世界和情境。

### 剧情结构

剧情应包括：

1. **世界/设定基础**
   - 我们在哪里？
   - 背景是什么？
   - 这个地方有什么有趣的？

2. **钩子**
   - 故事如何开始？
   - 为什么玩家参与其中？
   - 什么能立即抓住注意力？

3. **关键元素**（不预设结果）
   - 重要地点
   - 重要 NPC 或势力
   - 中心冲突或谜团
   - 潜在复杂情况

### 写作技巧

- 使用第二人称（直接称呼玩家）
- 生动地设定场景，但为玩家能动性留出空间
- 建立利害关系而不预定结果
- 包含感官细节以增强氛围

### 剧情示例

```
暴风雨的夜晚，你收到一个发黄的信封——没有邮票，没有寄件地址，
只有你的名字用优雅的字体写着。里面是一把钥匙和一张便条：
「遗产是你的，但债务必须偿还。在血月消失前来到黑木庄园。」

黑木庄园已经废弃五十年了，自从黑木家族在一次晚宴上消失。
当地人谈论窗户里的灯光，空荡荡房间里传出的音乐。这座庄园
藏着跨越三代人的秘密：一段禁忌之恋，一笔被盗的遗产，以及
一个被打断的仪式。

当你站在铁门前，雨水与领子上的锈迹混在一起时，
你意识到今晚不是唯一收到邀请的人。
```

## 阶段 3：任务定义

**目标**：定义清晰的 `mission_task`，给玩家目标和方向。

### 任务组成部分

1. **主要目标**
   - 玩家试图完成什么？
   - 保持足够开放以允许多种方法

2. **次要目标**（可选）
   - 丰富体验的额外成就
   - 要发现的秘密，要建立的关系

3. **互动指南**
   - 玩家应该如何与世界互动？
   - 有什么机制可用？

### 任务示例

**探索导向**：
```
探索黑木庄园，揭开家族失踪背后的真相。发现至少三个关于
庄园历史的隐藏秘密。决定是完成被打断的仪式、摧毁它的残余，
还是让过去埋葬。
```

**关系导向**：
```
在三方争夺城市控制权的政治紧张中周旋。通过谈判、间谍活动
或策略联姻建立联盟。你的选择将决定谁统治——谁陨落。
```

**生存导向**：
```
在隔离区生存七天，同时寻找解药原型。管理有限资源，避开
或对抗感染者，并决定幸存者中谁值得你的信任——以及你的物资。
```

## 阶段 4：注意与边界

**目标**：建立 `mission_plot_attention`，让 AI 与用户的期望保持一致。

### 要包括的内容

1. **基调指南**
   - 「保持氛围恐怖但不用 jump scares」
   - 「保持基调异想天开和略带荒诞」
   - 「避免对暴力的 graphic 描述」

2. **内容边界**
   - 「没有性内容或 NPC 的浪漫示好」
   - 「描述保持在 PG-13 等效水平」
   - 「避免 body horror 和 graphic 医疗细节」

3. **机制约束**
   - 「玩家选择应在3次互动内有可见后果」
   - 「绝不要强迫玩家进入战斗——他们总能谈判」
   - 「魔法应该感觉神秘，不像游戏机制」

4. **风格偏好**
   - 「使用现在时，第二人称视角」
   - 「每次回复以有意义的选择结束」
   - 「包含适合设定的感官细节」

### 注意示例

```
保持哥特式恐怖氛围，强调心理紧张而非 graphic 内容。
玩家能动性至关重要——绝不要强迫行动或替他们做选择。
后果应该感觉是他们决定的有机结果。将绑定角色（Victoria）
作为有自己议程的勉强盟友。她可以提供信息但不应该为玩家
解决问题。避免：graphic 暴力、jump scares、性内容、打破
第四面墙。使用：生动的描述、环境叙事、有意义的道德困境。
```

## 阶段 5：可选增强

### 角色绑定

如果用户希望 Agent 扮演特定角色：

1. 讨论角色概念
2. 要么：
   - 按名称/UUID 引用现有 TCP
   - 先引导用户创建新角色
3. 解释这个角色将在故事中如何运作

### 视觉上下文

`additional_prompts` 有助于为故事生成图像：

```
维多利亚式哥特庄园、雷暴、烛光、油画、
古董家具、蜘蛛网、1890年代美学、柔和色彩、戏剧性阴影
```

### 图片

如果用户想要视觉素材：
- 使用 `make_image` 生成头图/背景图
- 将 artifact UUID 传递给剧本创建

## 阶段 6：创建与验证

### 构建命令

一旦收集完所有字段：

```bash
neta-cli create_travel_campaign \
  --name "..." \
  --mission_plot "..." \
  --mission_task "..." \
  --mission_plot_attention "..." \
  --additional_prompts "..." \
  --tcp_uuid "..." \
  --status "PUBLISHED"
```

### 验证结果

立即调用 `request_travel_campaign` 以：
1. 确认成功创建
2. 向用户展示格式化结果
3. 高亮将在游玩模式中使用的 `custom_prompt`

## 要避免的常见陷阱

1. **过度脚本化**：不要写具体结果。设置情境，不是结论。

2. **任务混淆**：mission_plot 设定场景；mission_task 给出目标。不要混淆。

3. **模糊的注意**：「让它有趣」没用。要具体说明约束。

4. **太长**：虽然你可以写详细内容，但记住 AI 每次交互都会处理这个。保持专注。

5. **角色覆盖**：如果绑定 TCP，澄清 Agent 将扮演该角色，不是该角色是 NPC。

## 迭代完善

首次试玩后：
- 注意 AI 偏离意图的地方
- 使用 `update_travel_campaign` 收紧 plot_attention
- 如果玩家目标不清晰则调整任务
- 如果使用 TCP 则完善角色扮演

记住：最好的剧本是通过游玩演化的！
